# Tesseract — Pre-Governance Observation Engine PRD

**Product:** Faramesh Tesseract  
**Repo (planned):** `faramesh/tesseract` (or `faramesh tesseract` subcommand of Core)  
**License:** Open source (Apache 2.0) for self-use; cloud-hosted analysis via Horizon subscription  
**Relationship to Core:** Tesseract runs **before** Core enforcement is enabled. It answers: "what would governance look like for this agent?" Once you're ready, Tesseract generates the initial policy and you flip to enforcement mode.

---

## Problem Statement

The #1 adoption blocker for any enforcement primitive is fear of breakage. Engineers say: "I don't know what my agent does in detail. If I turn on governance, will it break my workflows? What rules do I even need?"

The answer is a pre-enforcement observation phase: run Tesseract alongside your agent in observe mode, record everything, analyze the behavioral patterns, generate a calibrated initial policy, then enable enforcement with confidence. This is the "Datadog agent install → data flows → you're bought in" moment for governance.

Without Tesseract, the path is: see demo → read docs → guess at policies → deploy → break production → disable governance in frustration. With Tesseract, the path is: instrument → observe → auto-generate policy → review → enforce.

## Core Features

### Observe Mode

Tesseract wraps your agent tools exactly like `govern()` does, but in non-blocking mode. Every tool call is recorded. Nothing is denied or deferred.

```python
# Step 1: Observe. Record everything. Enforce nothing.
from faramesh import tesseract

@tesseract.observe(agent_id="payment-bot", session="observe-session-001")
def stripe_refund(amount: int, customer_id: str) -> dict:
    return stripe.refunds.create(amount=amount, customer=customer_id)

# CLI equivalent:
# faramesh tesseract observe --agent payment-bot --output ./tesseract-data/
```

All observations are written to a local SQLite store: tool name, argument structural signatures, argument value distributions (numeric ranges, string length buckets, cardinality), call frequency, session context, principal context.

**Crucially: Tesseract does NOT store raw argument values.** It stores structural signatures + statistical summaries. This means it can generate governance policies without creating a PII store of your production data.

### Governance Readiness Score

After one observation window (default: 7 days), Tesseract computes a **Governance Readiness Score**:

```
faramesh tesseract score --agent payment-bot

  Governance Readiness Score: 67/100 ⚠️  Needs Attention
  ──────────────────────────────────────────────────────
  ✓ All tools visible and classifiable (23 distinct tools)
  ✓ Argument distributions stable over observation window
  ✗ 4 tools have no natural policy boundary (unbounded ranges)
  ✗ 2 high-entropy string arguments detected (potential secrets in args)
  ✗ Session budget not calibrated (no cost metadata on 8 tools)
  ⚠ 1 tool called with very high cardinality array args (potential bulk op)
  
  Recommendation: Review 6 flagged items before enabling enforcement.
  Run: faramesh tesseract report --agent payment-bot --format html
```

### Policy Generation

```bash
# Generate initial policy from 7 days of observations
faramesh tesseract generate \
  --agent payment-bot \
  --last 7d \
  --output policies/payment-bot-generated.yaml \
  --confidence-threshold 0.85

# Output: a real FPL policy file, ready for review
# Marks each rule with confidence score and evidence count
```

Generated policy example:
```yaml
# Generated by Faramesh Tesseract v1.0 from 7-day observation of payment-bot
# Observation window: 2026-02-21 to 2026-02-28 | 1,847 tool calls observed
# Review all rules marked ⚠ before enabling enforcement

faramesh-version: "1.1"
agent-id: payment-bot

rules:
  # Confidence: 0.97 | Evidence: 412 calls | Observed range: $0 - $487
  - id: tess-001
    description: "Generated: refund amount within observed range"
    match:
      tool: stripe/refund
      when: "args['amount'] <= 500"
    effect: permit

  # Confidence: 0.97 | Evidence: 0 calls above threshold
  - id: tess-002
    description: "Generated: refund above observed maximum requires approval"
    match:
      tool: stripe/refund
      when: "args['amount'] > 500"
    effect: defer
    reason: "Refund amount exceeds observed baseline maximum"

  # ⚠ Confidence: 0.71 | Evidence: 3 calls | Unbounded range observed
  - id: tess-003
    description: "Generated (LOW CONFIDENCE): mass email recipients"
    match:
      tool: send_email
      when: "len(args['recipients']) > 50"
    effect: defer
    reason: "Mass email detected - review recipients list"
```

### Transition to Enforcement

```python
# Step 2: Review generated policy, then flip mode
from faramesh import govern

# Before: observe mode
# @tesseract.observe(agent_id="payment-bot")

# After: enforce mode (one-line change)
@govern(agent_id="payment-bot", policy="policies/payment-bot-generated.yaml")
def stripe_refund(amount: int, customer_id: str) -> dict:
    return stripe.refunds.create(amount=amount, customer=customer_id)
```

### Continuous Drift Detection (Post-Enforcement)

After enforcement is enabled, Tesseract continues running in the background as a passive observer:

## Data Model

Tesseract stores observations in a separate SQLite database (never mixed with Core's DPR):

```sql
-- Tesseract observation store (separate from DPR)
CREATE TABLE observations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    agent_id TEXT NOT NULL,
    tool_id TEXT NOT NULL,
    arg_structural_sig TEXT NOT NULL,  -- shape fingerprint, no values
    
    -- Statistical summaries per numeric argument
    numeric_min REAL,
    numeric_max REAL,
    numeric_p50 REAL,
    numeric_p95 REAL,
    numeric_p99 REAL,
    
    -- String argument summaries  
    string_length_p95 INTEGER,
    string_cardinality INTEGER,        -- unique values count
    
    -- Array argument summaries
    array_length_p95 INTEGER,
    array_length_max INTEGER,
    
    -- Call metadata
    call_count INTEGER DEFAULT 0,
    session_count INTEGER DEFAULT 0,   -- distinct sessions this tool was called in
    first_seen TIMESTAMPTZ,
    last_seen TIMESTAMPTZ,
    observation_window_days INTEGER
);
```

**Privacy guarantee:** Tesseract never stores raw argument values. It stores only structural signatures and numeric summaries. A Tesseract observation store can be shared with Faramesh Labs for analysis support without leaking production data.

## Integration Points

- **Core:** Tesseract observation data is the seed for `faramesh policy-gen --from-observations`
- **Horizon:** Tesseract drift alerts surface in the Horizon fleet dashboard under "Behavioral Drift"
- **Hub:** Tesseract identifies when a built-in Hub pack would cover the observed tool patterns and suggests it
